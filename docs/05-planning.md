Planned additions and improvements to the O.C.D. project. Items are organized by priority and tracked with status. The **Token Preservation & Context Optimization** initiative takes priority over all other planned items.

## Token Preservation & Context Optimization

Highest-priority initiative. Goal: eliminate repetitive, high-token-cost work from the LLM loop through external automation. Every feature below replaces LLM reasoning or context with a deterministic, cacheable external tool. Projected savings: 23,000-32,000 tokens per typical session (~40-55% of non-user context).

### TP-1: Smart KB Injection Tool (Priority: Highest)

| Field | Detail |
| ---------- | --------------------------------------------------------------------------------------------------------------------------- |
| What | Replace naive KB context dump with intelligent relevance-ranked injection. `ocd kb query --relevant-to "<request>"` returns 3-5 most relevant articles + 50-token KB health card instead of full index. |
| Why | SessionStart currently injects up to 20,000 chars (~4,356 tokens) of undifferentiated KB index regardless of what the user is about to work on. Only 3-5 articles are typically relevant. |
| Est. Savings | ~3,500-4,000 tokens per session start |
| Status | Planned |

Acceptance criteria:

- `ocd kb query --relevant-to "auth redirect"` returns 3-5 ranked articles in \<2s
- SessionStart injects ≤8,000 chars (down from ≤20,000)
- `kb-index.json` auto-generated by `ocd-compile` with TF-IDF vectors for every article
- Results deterministic: same query → same articles → same order
- KB health card: `KB: 55 articles, 2 connections, last compiled 2026-04-21, 0 lint warnings`
- Zero external API calls in relevance path (pure local TF-IDF, no embeddings)

Integration points:

- `src/ocd/relevance.py` — new module: `score_articles(query, kb_index_json, top_k=3)`, `build_health_card()`
- `src/ocd/compile.py` — add `build_kb_index_json()` at end of compilation
- `src/ocd/hooks/session_start.py` — replace `build_context()` to call relevance engine
- `src/ocd/config.py` — add `KB_INJECTION_COUNT = 3`, `MAX_RELEVANT_CONTEXT_CHARS = 8000`
- `.agent/.state/kb-index.json` — new generated artifact

### TP-2: Lightweight Task Router (Priority: High)

| Field | Detail |
| ---------- | --------------------------------------------------------------------------------------------------------------------------- |
| What | External Python router replaces main LLM agent-selection reasoning. `ocd-route "<request>"` scores request against `manifest.json` keywords and returns optimal 1-3 agent names. |
| Why | Main LLM currently reads all 25 agent names/descriptions (~1,200 tokens always present) and reasons about which to call (~500-800 tokens per delegation). A keyword matcher does this in \<50ms with zero tokens. |
| Est. Savings | ~500-800 tokens per delegation + ~1,200 tokens from system prompt |
| Status | Planned |

Acceptance criteria:

- `ocd-route "find dead code"` → `dead-code-hunter`
- `ocd-route "security scan"` → `owasp-scanner`
- Multi-topic queries return up to 3 agents in relevance order
- `manifest.json` auto-generated from agent `.md` frontmatter by `ocd-compile --manifest`
- Router runs in \<100ms with zero API calls
- Claude Code system prompt references `ocd-route` for agent selection

Integration points:

- `src/ocd/router.py` — new module: `route_query(query, manifest, max_agents=3)`, `build_manifest()`
- `src/ocd/compile.py` — add `--manifest` flag
- `src/ocd/cli.py` — add `ocd route <query>` command
- `.claude/agents/manifest.json` — generated artifact (gitignored)

### TP-3: Standards-as-Reference System (Priority: High)

| Field | Detail |
| ---------- | --------------------------------------------------------------------------------------------------------------------------- |
| What | Extract Eight Standards text into versioned `standards.md` with hash reference. Agents receive only `ocd-standards:v1.0 [hash]` instead of full ~1,919-token text. System auto-injects full text on demand via `Read`. |
| Why | Every `/ocd` invocation loads the full Eight Standards (~7,678 chars). Most agent invocations never need the standards text. A hash reference costs ~15 tokens. |
| Est. Savings | ~1,919 tokens per invocation that doesn't need standards immediately; ~3,800-5,700 for 2-3 invocations per session |
| Status | Planned |

Acceptance criteria:

- `standards.md` contains full Eight Standards with YAML frontmatter (version + hash)
- `SKILL.md` references standards by version hash, doesn't embed full text
- SessionStart hook includes one-line standards reference in injected context
- Hash verified at session start (warn if mismatch)
- All 25 agent `.md` files that reference standards use hash reference format

Integration points:

- `.claude/skills/ocd/standards.md` — new file, extracted from current SKILL.md
- `.claude/skills/ocd/SKILL.md` — modified to reference by hash
- `src/ocd/hooks/session_start.py` — add standards reference line to context
- `src/ocd/compile.py` — add hash update for standards.md
- `src/ocd/config.py` — add `STANDARDS_FILE` path constant

### TP-4: Closed-Loop Fix Family (Priority: Medium)

| Field | Detail |
| ---------- | --------------------------------------------------------------------------------------------------------------------------- |
| What | Expand `ocd format` into `ocd fix-cycle`, `ocd lint-and-fix`, `ocd test-and-fix`, `ocd security-scan-and-patch`. Each runs full detect-fix-verify cycle externally, returning structured JSON. LLM says "run ocd-fix-cycle on X" and gets the result. |
| Why | Current fix loops cost ~500-1,300 tokens per iteration (read error → propose fix → apply → re-read error). Typical 2-3 iteration cycle = ~1,500-3,900 tokens. External command replaces entire loop with one tool call. |
| Est. Savings | ~2,000-5,000 tokens per fix cycle; ~6,000-15,000 for 3-5 cycles per session |
| Status | Planned |

Acceptance criteria:

- `ocd fix-cycle <file>` returns JSON: `{fixed: [...], remaining: [...], exit_code: 0|1}`
- `ocd lint-and-fix <path>` processes all matching files
- `ocd test-and-fix` only auto-fixes if prior tests were green
- `ocd security-scan-and-patch` categorizes findings into auto-fixable and manual-review
- All commands idempotent and deterministic
- Zero LLM calls in any fix command (pure external automation)
- Auto-fix limited to formatters + safe lint patches only; logic bugs require manual review

Integration points:

- `src/ocd/fix.py` — new module: `FixResult` dataclass, `fix_cycle()`, `lint_and_fix()`, `test_and_fix()`, `security_scan_and_patch()`
- `src/ocd/cli.py` — add `fix-cycle`, `lint-and-fix`, `test-and-fix`, `security-scan-and-patch` subcommands
- `src/ocd/format.py` — expose `FORMATTERS` registry as public API
- `src/ocd/hooks/lint_work.py` — extract or import `LINTERS` registry for reuse
- `pyproject.toml` — add 4 new console script entry points

### TP-5: Pre-compiled Agent Manifest + Session State Card (Priority: Medium)

| Field | Detail |
| ---------- | --------------------------------------------------------------------------------------------------------------------------- |
| What | Part A: Generate `manifest.json` (~4 KB) at session start, replacing 97 KB of agent `.md` definitions. Part B: Auto-generate 200-300 token session state card after every file modification, replacing full history replay after compaction. |
| Why | All 25 agent files total 97 KB (~24,286 tokens). After compaction, the LLM loses most context and must reconstruct state. A 300-token session card captures essential state: what changed, what passed, what's pending. |
| Est. Savings | ~23,000 tokens from manifest; ~15,000-20,000 per post-compaction context rebuild |
| Status | Planned |

Acceptance criteria:

- `manifest.json` generated at session start from `.claude/agents/*.md` frontmatter
- Manifest contains `name`, `summary`, `keywords`, `tools`, `scope_hash` per agent
- Manifest total size ≤4 KB
- Session card auto-updated on every `Write` or `Edit` tool use
- Session card contains: files modified, test results, lint status, KB health, current task, open issues
- Session card size ≤300 tokens (~1,200 chars)
- `SessionStart` hook injects manifest reference + last session card

Integration points:

- `src/ocd/router.py` — `build_manifest()` (shared with TP-2)
- `src/ocd/session_card.py` — new module: `update_session_card()`, `build_session_card()`
- `src/ocd/hooks/session_start.py` — inject manifest ref + session card
- `src/ocd/hooks/format_work.py` — call `update_session_card()` after formatting
- `src/ocd/config.py` — add `SESSION_CARD_FILE`, `MANIFEST_FILE`, `MAX_SESSION_CARD_CHARS = 1200`
- `.agent/.state/manifest.json`, `.agent/.state/session-card.md` — generated artifacts

### TP Risks & Mitigation

| Risk | Mitigation |
| --- | --- |
| TF-IDF relevance poor for short queries | Fallback to 3 most-recently-updated articles when confidence below threshold |
| `kb-index.json` stale after updates | Session start checks hash against current articles, triggers re-index on mismatch |
| Agent manifest keywords insufficient for routing | Include `scope_summary` for disambiguation; log routing decisions; allow `ocd-route --force <agent>` |
| Session card grows too large | Hard cap at 1,200 chars, FIFO eviction of oldest entries |
| Fix-cycle makes incorrect auto-fixes | Only auto-fixes via formatters (already safe); lint auto-fix requires explicit `--fix` flag |
| Feature scope creep | Strict acceptance criteria per feature; each new module ≤300 lines; no new config knobs without measurement justification |

Total new surface area: 4 Python modules (`relevance.py`, `router.py`, `fix.py`, `session_card.py`), 4 generated artifacts (all gitignored in `.agent/`), 5 new CLI entry points. All generated, all deterministic, all zero-dependency beyond existing `pyproject.toml`.

## Resolved Decisions

### Agent Architecture: Task-Driven (Decided)

Agents are **task-driven** — each does one focused job. Role-based agents were rejected because they conflate concerns, have broader surface area, and are harder to test. The task-driven model aligns with Minimal Surface Area and keeps each agent composable and precise.

The original role-based proposals (frontend-dev, backend-dev, devops-engineer, security-auditor) were replaced with task-driven equivalents: accessibility-auditor, api-contract-auditor, dockerfile-auditor, and owasp-scanner.
